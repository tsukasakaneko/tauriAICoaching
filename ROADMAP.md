# 開発ロードマップ(優先度付きタスク一覧)

**前提となる戦略方針**(2026-07 コードベース監査に基づく):

> 強みは「AI が分析すること」ではなく、**「ゼロ操作で録れて、先週の自分と比較され、改善が数字で見えること」**。
> LLM コーチング単体は ChatGPT で代替可能で堀にならない。防御力があるのは
> ①蓄積された個人の試合履歴と改善ループ(乗り換えコスト)、②日本語特化の知識ベース、③ゼロ操作 UX の 3 つ。
> YOLOv8 モデル訓練は最も工数が大きく未着手のため後回しにし、まず **Riot ローカル API + 閉ループ**で
> 「動く収益マシン」を最小構成で完成させる。映像解析は API で取れないポジショニング領域に絞って後から積む。

**BUSINESS_PLAN.md からの変更点**:

| 項目 | 旧計画 | 本ロードマップ |
|------|--------|--------------|
| クレジット失効 | 購入から1年 | **6ヶ月**(資金決済法・前払式支払手段の適用除外要件) |
| 固定費 | 60,000円/月(Render 40,000円) | **約25,000円/月**(backend-remote は軽量プロキシのため最小プランで十分) |
| BEP | 月額約81名 | **約30名**(固定費削減による) |
| 転換率 DL→有料 | 5%→15% | **1〜3%** で逆算(有料270名 ≒ 累計9,000DL が必要) |
| YOLOv8 解析 | 核心機能(実装済み扱い) | **P2 に降格**(モデル未訓練のため。Riot ローカル API で代替) |
| 分析履歴・改善トラッキング | Phase 2(リテンション施策) | **P1(プロダクトの核)に昇格** |

優先度の定義:
- **P0** — 収益化ブロッカー。これが終わるまで集客・マーケ費投入をしない
- **P1** — 差別化の核(閉ループ)。P0 完了後ただちに着手
- **P2** — グロース・拡張。黒字化後

## P0 — 収益化ブロッカー

### P0-1. [#20] サーバー側クレジット台帳への移行

- **目的**: クレジット残高・使用済みノンスが Tauri ローカルストアのみ(`src-tauri/src/commands/license.rs`, `src-tauri/src/commands/ai.rs`)のため、ストア編集で残高改ざん・再インストールで VCREDIT キー再利用が可能。キーがメールに紐付いておらず共有・転売も防げない。収益保護の最優先事項。
- **内容**:
  - `backend-remote` に SQLite(better-sqlite3)を追加し、`licenses`(key_hash, email, tier, credits_granted, activation_count, max_activations, expires_at)と `credit_ledger`(license_id, delta, reason, created_at)テーブルを作成
  - `POST /license/activate`(キー検証+回数制限+残高付与)と `POST /credits/consume`(サーバー側デクリメント)を新設
  - Stripe webhook(`backend-remote/server.js`)でキー発行時に DB へ記録(現状は発行+メール送信のみで記録なし)
  - `commands/license.rs`・`ai.rs` をサーバー照会に変更。ローカルストアは表示用キャッシュに格下げ
- **完了条件**: ストアファイル編集・再インストールでクレジットが復元できない。同一キーの2台目以降アクティベートが拒否される
- **規模**: 大(3〜5日)

### P0-2. [#21] 課金モデルの一本化

- **目的**: `backend/routes/coaching.js` の日次制限(無料5回/有料20回)と BUSINESS_PLAN のクレジット制が併存し不整合。ユーザーが購入したクレジットと実際の制限が一致しない。
- **内容**: 「無料3回/日+有料はクレジット消費(手動1・自動録画2)」に統一。`coaching.js` の `DAILY_LIMIT_FREE/PAID`、`backend-remote/server.js` の `DAILY_LIMIT=30` を整理し、P0-1 の消費 API に接続
- **完了条件**: 無料/有料の制限動作が BUSINESS_PLAN の価格表と 1:1 で一致する
- **規模**: 小(0.5〜1日)

### P0-3. [#22] プロンプト構築の一元化

- **目的**: プロンプト構築が 3 箇所に重複(`backend/routes/coaching.js`・`src-tauri/src/prompt_builder.rs`・`backend-remote/server.js`)。リモート版のみ知識ベース(`agent_knowledge.toml`)注入がなく、**課金ユーザーの方が質の低いプロンプトになる逆転**が発生中。
- **内容**: Tauri 側(`prompt_builder.rs`)で構築した system/user プロンプトをリモートへ送る形に統一し、Node 側の重複実装を削除。あわせて `estimateLateRoundDeaths` 等の推定値をプロンプトから除去するか「推定」と明記(AI が架空の根拠で断定するのを防ぐ)
- **完了条件**: プロンプト構築ロジックが 1 箇所になり、ローカル/リモートで同一品質のレポートが出る
- **規模**: 中(1〜2日)

### P0-4. [#23] クレジット失効を6ヶ月に変更

- **目的**: 有効期限1年の前払クレジットは資金決済法の前払式支払手段に該当し得る(未使用残高1,000万円超で供託義務等)。**6ヶ月以内なら適用除外**。
- **内容**: `src-tauri/src/license.rs` の期限判定、`backend-remote/server.js` の `expiryForTier`、`BUSINESS_PLAN.md`、`src/components/UpgradeModal.tsx` の表記(「1年有効」→「6ヶ月有効」)を更新
- **完了条件**: 発行キー・UI・計画書のすべてが6ヶ月失効で整合
- **規模**: 小(0.5日)

### P0-5. [#24] 法務ページの整備

- **目的**: EC として特定商取引法の表記が販売開始前に必須。返金ポリシー・プライバシーポリシーも同様。
- **内容**: 特商法表記/返金ポリシー/プライバシーポリシーの静的ページを作成(P1-12 の購入 LP に掲載。先行してリポジトリ内に原稿を用意)
- **完了条件**: 3 文書が LP から到達可能
- **規模**: 小(0.5〜1日、法テラス等での確認は別途)

### P0-6. [#25] アプリ名から VALORANT 商標を外す

- **目的**: 製品名への Riot 商標使用はサードパーティポリシー違反。現状 UI に「VALORANT AI Coaching」表記(`src/components/ReplayScreen.tsx` ほか各画面の brand 表記)。
- **内容**: 「<製品名> for VALORANT」形式へ変更。`src-tauri/tauri.conf.json`(productName)、各画面のブランド表記、`backend-remote/server.js` のメール文面・差出人ドメインを更新
- **完了条件**: 製品名単体に Riot 商標が含まれない。試合中のリアルタイム助言機能を今後も作らない方針を README 等に明記
- **規模**: 小(0.5日)

### P0-7. [#26] インフラ費の最小化

- **目的**: 重い処理(録画・解析)は全てユーザーのローカルで動くため、`backend-remote`(約450行の Express プロキシ)に月40,000円は過大。固定費 60,000円→約25,000円で BEP が月額約81名→約30名に下がる。
- **内容**: `render.yaml` を最小プラン(Starter 相当)に変更。P0-1 の SQLite 永続化にはディスク付きプランかマネージド Postgres 最小構成を選定
- **完了条件**: 月額インフラ費 5,000円以下で本番稼働
- **規模**: 小(0.5日)

## P1 — 差別化の核(閉ループ)

### P1-8. [#27] 分析履歴画面

- **目的**: `match_sessions`・`video_analysis_json` は保存済み(`backend/db.js`)だが閲覧手段がない。履歴は乗り換えコスト=リテンションの土台。
- **内容**: 履歴一覧 API(`backend/routes/`)+ 履歴画面(React)。セッションごとに日時・マップ・KDA・レポートへのリンクを表示
- **完了条件**: 過去の全セッションとレポートを一覧から開ける
- **規模**: 中(1〜2日)

### P1-9. [#28] 改善トラッキング(前回比)

- **目的**: 「7日間プランを出して終わり」を閉ループ化する。プラン → 次の試合を自動計測 → 「HS率 23%→28% に改善」という人間コーチ相当のフィードバックがチャーン対策の本命。
- **内容**: 直近セッションの指標(HS率・KDA・aggressiveness・dominantZone)を取得してプロンプトに注入(`prompt_builder.rs`)。レポートに「前回比」セクションを追加(`ReportScreen.tsx`、JSON スキーマ拡張)
- **完了条件**: 2 回目以降の分析で前回比が数値で表示され、AI のアドバイスが前回の課題を参照する
- **規模**: 中(2〜3日)

### P1-10. [#29] Riot ローカルクライアント API 連携

- **目的**: KDA・試合結果は Riot クライアントのローカル API(lockfile 経由)で取得可能。YOLO なしで「ゼロ入力」体験の 8 割を実現し、未訓練の YOLO パイプラインへの依存を断つ。
- **内容**: lockfile 読み取り+ローカル API クライアントを `backend/services/` に追加し、試合終了検知・KDA・エージェント・マップを自動取得。`videoAnalyzer.js` の統計系(killfeed OCR 等)はフォールバック扱いに
- **完了条件**: 試合終了後、手入力ゼロで KDA・マップ・エージェント入りのレポートが生成される
- **規模**: 大(3〜5日)

### P1-11. [#30] レポート PNG 共有

- **目的**: 「AI 分析レポートのスクショが X で拡散される」がオーガニック CAC<300円 の生命線。共有を 1 クリック化する。
- **内容**: レポート画面を画像化(html-to-image 等)して保存/X 共有ボタンを追加。画像に製品名とダウンロード URL を焼き込む
- **完了条件**: 1 クリックでブランド入りレポート画像が生成される
- **規模**: 小(1日)

### P1-12. [#31] 購入ランディングページ

- **目的**: 現状アプリ内 `UpgradeModal` → Stripe Checkout のみで、Web からの流入を受ける先がない。
- **内容**: 静的 LP(GitHub Pages/Vercel)。価格表・機能紹介・Stripe Checkout 導線・P0-5 の法務ページ掲載・ダウンロードリンク
- **完了条件**: 広告/SNS から LP 経由で購入完了まで到達できる
- **規模**: 中(1〜2日)

## P2 — グロース・拡張(黒字化後)

### P2-13. リファラルプログラム
紹介者に +5 クレジット。既存の VCREDIT キー発行(`src-tauri/src/bin/keygen.rs`・webhook 発行フロー)を転用。

### P2-14. 知識ベース拡充
`agent_knowledge.toml` をマップ別セットアップ・パッチメタ追随まで拡張。日本語コーチング品質を競合が真似しにくい資産に育てる。

### P2-15. 映像解析の再スコープ
API で取れない領域(ミニマップ上のポジショニング履歴・ピークの癖・アビリティ使用タイミング)に限定。着手前に **YOLO 訓練 vs キーフレームを Claude vision に直接投げる方式のコスト試算スパイク**を実施し、安い方を採用。

### P2-16. B2B2C チャネル検討
eスポーツ専門学校・高校部活・ゲーミングスクール向けのスクール単位ライセンス。月20万円目標なら法人 2〜3 件で到達可能。
